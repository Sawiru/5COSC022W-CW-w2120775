# CSA Coursework 
Repository for the 5COSC022W Client-Server Architectures coursework (2025/26). This project implements a high-performance "Smart Campus" RESTful API using strictly JAX-RS and in-memory data structures.

## Part 1 — Q1: JAX-RS Resource Lifecycle
By default JAX-RS uses request-scoped lifecycle, meaning a brand new instance of each resource class is instantiated for every incoming HTTP request and discarded immediately after the response is sent. This is the opposite of a singleton where one shared instance handles all requests. The request-scoped default means instance variables cannot be used to store persistent data since they are reset on every request. To maintain state across requests, the DataStore class uses static ConcurrentHashMap fields which exist at the class level for the entire application lifetime. ConcurrentHashMap is chosen over a regular HashMap because JAX-RS handles concurrent requests on multiple threads simultaneously, and HashMap is not thread-safe — simultaneous writes from two threads can corrupt the internal structure entirely. ConcurrentHashMap uses internal segment locking to allow safe concurrent reads and writes without requiring manual synchronized blocks, preventing race conditions and data loss.

## Part 1 - Q2: What is HATEOAS?
HATEOAS (Hypermedia as the Engine of Application State) is the principle that API responses should include hypermedia links guiding the client to related actions and resources, rather than requiring clients to construct URLs themselves. For example a room response would include links such as GET /api/v1/rooms/LIB-301/sensors to view its sensors and DELETE /api/v1/rooms/LIB-301 to remove it. This makes the API self-documenting — clients discover available actions at runtime by following links, just like a browser navigates a website without needing to know URLs in advance. Compared to static documentation, HATEOAS is superior because static docs go out of date when URLs change, forcing developers to update every client manually. With hypermedia links, clients automatically adapt to URL changes since they always follow what the server provides rather than hard-coding paths.

## Part 2 — Q3: Returning full objects vs IDs in a list
When returning a list of rooms, returning full objects gives the client everything they need in a single request which is convenient but costly in terms of bandwidth, especially when there are hundreds of rooms each with many fields. Returning only IDs is very lightweight but forces the client to make an additional GET request for every single room they want details about, which can result in dozens of extra network calls. The recommended approach for most APIs is to return full objects in list responses but keep them lean by excluding deeply nested relationships. For example a room list should include the room's own fields but not embed the full sensor objects inside it — just the sensor IDs is sufficient at that level.

## Part 2 -  Q4: Is DELETE idempotent?
Yes, the DELETE operation is idempotent in this implementation. Idempotency means that sending the same request multiple times produces the same server state as sending it once. If a client sends DELETE /api/v1/rooms/LIB-301 and the room exists, it gets deleted. If the same request is sent again, the room no longer exists and the server returns a 404 Not Found response. Although the HTTP response code differs between the first and second call, the server state is identical after both — the room is gone either way and no data corruption or unintended side effect occurs. This makes DELETE safe to retry in cases of network failure.

## Part 3 - Q5: What happens if wrong Content-Type is sent to @Consumes endpoint
The @Consumes(MediaType.APPLICATION_JSON) annotation tells JAX-RS that the POST endpoint only accepts requests with a Content-Type header of application/json. If a client sends data with a different Content-Type such as text/plain or application/xml, JAX-RS intercepts the request before it even reaches the method body and automatically returns an HTTP 415 Unsupported Media Type response. The method is never executed. This is handled entirely by the JAX-RS runtime, meaning no manual checking is needed inside the code — the framework enforces the contract declared by the annotation.

## Part 3 — Q6: @QueryParam vs path segment for filtering
Using @QueryParam for filtering, as in GET /api/v1/sensors?type=CO2, is considered superior to embedding the filter in the path such as GET /api/v1/sensors/type/CO2 for several reasons. Query parameters are optional by nature, meaning the same endpoint handles both the filtered and unfiltered case cleanly without needing two separate methods. They also clearly communicate intent — path segments represent resource identity and hierarchy while query parameters represent optional modifiers on a collection. Additionally multiple filters can be combined naturally with query parameters such as ?type=CO2&status=ACTIVE whereas doing this with path segments would require a complex and confusing URL structure. Query parameters are also the standard convention understood by every HTTP client and caching layer.

## Part 4 — Q7: Sub-Resource Locator pattern benefits
The Sub-Resource Locator pattern works by having a method in the parent resource class return an instance of a separate child resource class rather than handling the nested path directly. In this case SensorResource delegates anything under /{sensorId}/readings to a dedicated SensorReadingResource class. The main benefit is separation of concerns — each class has one clear responsibility. If every nested endpoint was defined in one massive resource class the file would become extremely difficult to read, test, and maintain as the API grows. By splitting into sub-resources each class stays small and focused. It also makes the code more reusable since SensorReadingResource can in theory be instantiated from multiple parent resources if needed. This mirrors good object-oriented design principles and reflects how large real-world APIs are structured in production systems.

## Part 5 — Q8: Why 422 is more semantically accurate than 404 for a missing roomId reference
HTTP 404 Not Found is intended to signal that the URL or endpoint being requested does not exist on the server. However when a client sends a POST request to /api/v1/sensors with a roomId that does not exist in the system, the URL /api/v1/sensors is completely valid and reachable — the problem is not with the endpoint itself but with the content inside the request body. HTTP 422 Unprocessable Entity is more semantically accurate in this scenario because it signals that the server successfully received and understood the request, the JSON was syntactically well-formed, but the payload failed a business logic validation rule — specifically that the referenced roomId does not exist. Using 404 here would mislead the client into thinking they called a wrong or non-existent URL, when in reality the URL is correct and the issue lies inside their JSON payload. The 422 status code communicates precisely what went wrong, making it far easier for client developers to debug and correct their requests.

## Part 5 - Q9: Stack trace security risks 
Exposing Java stack traces to external consumers presents serious cybersecurity risks. First, stack traces reveal the exact internal package and class structure of the application, telling an attacker the precise frameworks and libraries in use along with their versions. This allows the attacker to search public vulnerability databases like CVE for known exploits targeting those exact versions. Second, stack traces can expose internal file system paths such as where the application is deployed, which can be used in path traversal attacks. Third, they reveal application logic flow — showing exactly which method called which, which an attacker can use to identify points of weakness or craft targeted inputs. The GlobalExceptionMapper solves this entirely by intercepting all Throwable instances before they reach the HTTP response layer and returning only a generic 500 message, ensuring the internal architecture is never disclosed to external consumers.
