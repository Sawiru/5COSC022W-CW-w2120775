# CSA Coursework 
Repository for the 5COSC022W Client-Server Architectures coursework (2025/26). This project implements a high-performance "Smart Campus" RESTful API using strictly JAX-RS and in-memory data structures.

## Part 1 — Q1: JAX-RS Resource Lifecycle
By default, JAX-RS creates a brand new instance of each resource class for every incoming HTTP request. This is called per-request scope. Because of this, any data stored as an instance variable inside a resource class would be lost the moment the request finishes. To persist data across requests, the DataStore class uses static ConcurrentHashMap fields, which live at the class level and survive the entire lifetime of the application. ConcurrentHashMap is used instead of a regular HashMap because multiple HTTP requests can arrive simultaneously from different threads, and a regular HashMap is not thread-safe — concurrent writes to it can corrupt the data structure entirely. ConcurrentHashMap handles this internally by allowing safe concurrent access without needing manual synchronization.

## Part 1 - Q2: What is HATEOAS?
HATEOAS stands for Hypermedia as the Engine of Application State and is considered the highest level of RESTful API maturity. The core idea is that API responses include not just data but also links to related actions and resources. For example, a response returning a room object would also include links such as a link to view its sensors or delete it. This benefits client developers significantly because they do not need to memorise or hard-code URLs — they simply follow the links provided in each response, similar to how a browser navigates a website. It also makes the API more resilient to change since if a URL structure changes, clients following hypermedia links automatically adapt rather than breaking.

## Part 2 — Q3: Returning full objects vs IDs in a list
When returning a list of rooms, returning full objects gives the client everything they need in a single request which is convenient but costly in terms of bandwidth, especially when there are hundreds of rooms each with many fields. Returning only IDs is very lightweight but forces the client to make an additional GET request for every single room they want details about, which can result in dozens of extra network calls. The recommended approach for most APIs is to return full objects in list responses but keep them lean by excluding deeply nested relationships. For example a room list should include the room's own fields but not embed the full sensor objects inside it — just the sensor IDs is sufficient at that level.

## Part 2 -  Q4: Is DELETE idempotent?
Yes, the DELETE operation is idempotent in this implementation. Idempotency means that sending the same request multiple times produces the same server state as sending it once. If a client sends DELETE /api/v1/rooms/LIB-301 and the room exists, it gets deleted. If the same request is sent again, the room no longer exists and the server returns a 404 Not Found response. Although the HTTP response code differs between the first and second call, the server state is identical after both — the room is gone either way and no data corruption or unintended side effect occurs. This makes DELETE safe to retry in cases of network failure.

## Part 3 - Q5: What happens if wrong Content-Type is sent to @Consumes endpoint
The @Consumes(MediaType.APPLICATION_JSON) annotation tells JAX-RS that the POST endpoint only accepts requests with a Content-Type header of application/json. If a client sends data with a different Content-Type such as text/plain or application/xml, JAX-RS intercepts the request before it even reaches the method body and automatically returns an HTTP 415 Unsupported Media Type response. The method is never executed. This is handled entirely by the JAX-RS runtime, meaning no manual checking is needed inside the code — the framework enforces the contract declared by the annotation.

## Part 3 — Q6: @QueryParam vs path segment for filtering
Using @QueryParam for filtering, as in GET /api/v1/sensors?type=CO2, is considered superior to embedding the filter in the path such as GET /api/v1/sensors/type/CO2 for several reasons. Query parameters are optional by nature, meaning the same endpoint handles both the filtered and unfiltered case cleanly without needing two separate methods. They also clearly communicate intent — path segments represent resource identity and hierarchy while query parameters represent optional modifiers on a collection. Additionally multiple filters can be combined naturally with query parameters such as ?type=CO2&status=ACTIVE whereas doing this with path segments would require a complex and confusing URL structure. Query parameters are also the standard convention understood by every HTTP client and caching layer.
